PodBlogger

UX analysis and redesign of podcast listening apps.

BRIEF:

Research and design a new social media service which is focused on people sharing spoken word audio recordings (podcasts).

29 April 2020
UNSW UX Design

USER RESEARCH

I started by interviewing a number of people about their current podcast listening habits and frustrations. The most used podcast app was Apple Podcasts, even though users were often frustrated with its functionality. I discovered that the exploration of new content is a major pain point, with many users using external apps to find new podcasts. In relation to this, interviewees stated that they would like to see features which allow reviewing, filtering and suggested content.


COMPARITIVE ANALYSIS

Filtering podcasts:

In current podcast apps there is no capability to filter searches. For example in the Apple Podcast app, the only input the user has in searching is the search text. This is limiting as the user has to scroll through endless podcasts to discover new content. A non podcast app that handles this task well is ASOS. Users are able to sort and filter through search results . A large number of filtering criteria is provided. Recent searches are saved to increase ease of use.


Reviewing podcasts:

Current apps have some functionality in rating and reviewing podcasts. Apple podcasts allow users to leave a written or numerical rating. But podcasts cannot be sorted according to these. The Depop app has both forms of review, yet like apple, you cannot filter based on these reviews.


PERSONAS




USER FLOWS

Filtering podcasts:

When filtering a search, the user opens the Podblogger app and navigates to the search page. Then the user can select a category, enter text into the search bar or select the filter icon. On the filter page, the user selects filters such as time and rating then searches. The user scrolls through the page and selects a podcast to listen to.



Reviewing podcasts:

When leaving a review, the user opens the app and selects a podcast show. Then they can select the rate button or select a specific episode. Once selecting a specific episode, they can listen to the podcast first or rate the podcast. On the rating page the user rates the podcast on a variety of criteria. The user submits the rating and can return to the previous page.




ANALYSIS


The circular layout allows the user to view podcasts without scrolling endlessly. Some buttons on the screens have a clearer affordance such as the "Submit" button on the review page. Yet the rating button on the play page is less obvious as it is plain text. The filter page is cluttered with irrelevant data, which does not follow aesthetic and minimalist design. There is no way to view recent searches, which requires recall rather than recognition.


USER TESTING

Through user testing, the main areas of improvement were: The location of the search navigation icon - users didn't expect the icon to be the first option on the right, criteria for the ratings was confusing and unnecessary, buttons were too small and not easily clickable, ability to leave a written review..


USER FLOWS ITERATION 2

Filtering podcasts:

For the filtering user flow, the main changes made were: moving the search navigation to the second option on the right, hiding the filtering options within dropdown menus, and saving the users recent searches.


Reviewing podcasts:

For the review functionality, the main changes made were: the option to leave a written review or just numerical and a separate reviews page where the user can view reviews made by other users.


ANALYSIS ITERATION 2

Important buttons are now larger and clearer which reduces confusion for the user. When writing the reviews, placeholder text indicates what to write in each box. The size indicates the expected amount of text to be written. The filter page is now less cluttered with filters hidden away in drop down menus. The user can now view their recent searches when clicking on the search bar.
